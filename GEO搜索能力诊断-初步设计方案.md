# GEO 搜索能力诊断：初步设计方案

# 总览：执行步骤与讨论路线图

## 目标

给定一个开源社区及期望搜索的话题，自动完成主流 AI 平台的 GEO 搜索能力判断，并给出社区文档的改进建议。

## 系统架构

采用 **skill chain + AGENT.md 编排**，纯 CLI 驱动，通过 Claude Code 自动执行。

```
AGENT.md（工作流编排）
  │
  ├── Step 1: keyword-generator skill
  │     输入：社区名称 + 种子关键词
  │     输出：questions.json + questions.md
  │     ⏸ 人工审阅检查点
  │
  ├── Step 2: platform-sampler skill
  │     输入：questions.json
  │     输出：responses.json + responses.md
  │
  ├── Step 3: scoring-engine skill
  │     输入：responses.json
  │     输出：scores.md（评分报告）
  │
  └── Step 4: improvement-advisor skill
        输入：scores.md
        输出：suggestions.md（改进建议）
```

数据流：JSON 在 skill 之间传递，Markdown 供人类审阅。

## 执行步骤

---

### 步骤一：生成问题集（keyword-generator skill）

**目标**：输入社区名称和种子关键词，输出覆盖两个场景的问题集。

**已确定方案**：

覆盖两个场景：
- **了解阶段**：用户不知道社区，搜索行业大类问题（如"主流深度学习框架有哪些"）→ 社区应被提及
- **使用阶段**：用户已知社区，搜索具体使用问题（如"MindSpore 安装教程"）→ 回答应准确

问题来源：手动输入 + 三条自动生成路径

```
手动输入：社区运营编写 manual-questions.md → skill 自动解析为结构化问题

路径一：行业问题发现（了解阶段）
  社区名称 → LLM 确定领域层级（行业/细分/定位/竞品）
           → 按用户意图分类生成问题（认知/选型/趋势/场景）
           → 用竞品做反向扩展

路径二：使用问题提取（使用阶段）
  Issue（Gitee/GitHub API）→ 过滤纯 Bug → 聚类 → LLM 改写为自然语言问题
  论坛帖子（API 或爬取）  → 按热度排序 → LLM 改写为自然语言问题

路径三：AI 平台反向提取（使用阶段）
  问多个 AI 平台"关于{社区}最常被问到的问题" → 取交集
```

合并（手动 + 三条路径）→ 语义去重 → 分类标注 → 输出 `questions.json` + `questions.md`

**人工审阅检查点**：生成后暂停，等待人工筛选反馈。反馈保存到 `feedback-rules.md`，后续自动融入生成 prompt（学习循环）。

**MVP 数量**：30-40 个问题（可根据结果调整）。

**待讨论问题**：

| 编号 | 问题 | 状态 |
|------|------|------|
| 1.1 | Issue 数据范围：只看主仓库还是覆盖所有子项目？ | 待讨论 |
| 1.2 | MindSpore 论坛是否有公开 API？若无，是否接受爬取？ | 待调研 |

详细设计见 → 第二部分（第六~八节）、第九节

### 步骤二：多平台搜索采样（platform-sampler skill）

**目标**：调用各 AI 平台 API，用问题集采样，采集原始回答。

**已确定方案**：
- MVP 平台：Perplexity + ChatGPT + DeepSeek
- 统一接口，各平台各自实现适配器
- API Key 通过环境变量管理
- 保留 AI 回答原文用于后续评分和人工复核
- 详细平台分析见 → 第一部分（第一~五节）

**数据模型**：

```json
{
  "question_id": "q_001",
  "platform": "perplexity",
  "query": "主流深度学习框架有哪些",
  "timestamp": "2026-03-10T08:00:00Z",
  "raw_response": "...",
  "citations": ["url1", "url2"],
  "mentions_community": true,
  "competitors_mentioned": ["TensorFlow", "PyTorch"]
}
```

**输入**：`questions.json`
**输出**：`responses.json` + `responses.md`

**待讨论问题**：

| 编号 | 问题 | 状态 |
|------|------|------|
| 2.1 | Perplexity API 的 `search_recency_filter` 参数如何设置？ | 待调研 |
| 2.2 | ChatGPT API web search 返回的引用来源格式解析 | 待调研 |
| 2.3 | DeepSeek 无联网搜索，如何评估？仅测内置知识覆盖还是接入第三方搜索？ | 待讨论 |

### 步骤三：诊断分析（scoring-engine skill）

**目标**：对采样结果进行 LLM 结构化评分和对比分析，输出评分报告。

**已确定方案**：
- 评分体系：提及分(0-3) + 引用分(0-3) + 准确分(0-2) + 排名分(0-2) = 总分(0-10)
- 综合评分：Perplexity×0.35 + ChatGPT×0.40 + DeepSeek×0.25
- LLM 逐条评分，输出结构化 JSON（含四维分数和理由）
- 聚合分析：按平台、按关键词、竞品对比
- 内容缺口识别：提及分=0 → 缺口，引用分<2 → 引用缺失，准确分=0 → 信息过时
- 详细评分体系见 → 第三部分（第十~十一节）

**输入**：`responses.json`
**输出**：`scores.json` + `scores.md`（评分报告）

**待讨论问题**：

| 3.2 | 准确率判断是否需要提供社区官方文档作为参考上下文？ | 待讨论 |
| 3.3 | 评分的 LLM Prompt 模板如何设计？ | 待设计 |
| 3.4 | 评分结果是否需要人工抽检校准？ | 待讨论 |

### 步骤四：改进建议（improvement-advisor skill）

**目标**：根据评分结果，生成分优先级的可执行改进建议。

**已确定方案**：
- LLM 根据评分 + 原始回答 + 缺口类型生成建议
- 改进建议按 P0-P2 分优先级
- 建议去重（相似建议合并）
- 详细建议格式见 → 第三部分（第十二节）

**优先级规则**：
- **P0**：评分 0-3 且为高优先级问题 → 紧急补充内容
- **P1**：评分 4-6 → 优化现有内容
- **P2**：评分 7-8 → 微调优化
- 评分 9-10 → 无需行动，持续监控

**输入**：`scores.json` + `responses.json`
**输出**：`suggestions.md`（改进建议报告）

# 第一部分：主流 AI 搜索平台分析

## 一、平台分类

### 1.1 专用 AI 搜索引擎

| 平台 | 典型用户特征 | 判断依据 |
|------|-------------|---------|
| Perplexity | 技术从业者、研究者、英文信息消费者 | 产品定位为"答案引擎"，无中文界面，用户需有主动搜索习惯且偏好结构化引用 |
| Google AI Overview (SGE) | 最广泛的通用搜索用户 | 嵌入 Google 搜索结果页，用户无需主动选择，覆盖所有 Google 用户 |
| Bing Copilot | Windows/Edge 生态用户、企业用户 | 绑定 Edge 浏览器和 Windows 系统，企业 M365 用户自动接入 |
| 秘塔搜索 (Metaso) | 国内学术/知识型用户 | 主打"没有广告的搜索"，学术搜索功能突出，在知识社区传播 |
| 天工 AI 搜索 | 国内普通用户 | 昆仑万维推出，国内搜索市场补充 |

### 1.2 带搜索能力的 AI 助手

| 平台 | 典型用户特征 | 判断依据 |
|------|-------------|---------|
| ChatGPT | 全球最大 AI 用户群，偏技术和知识工作者 | MAU 超 2 亿，搜索功能默认开启，用户以提问方式获取信息 |
| 豆包 | 国内年轻用户、抖音生态用户 | 字节系产品，抖音导流，用户偏年轻化和泛娱乐 |
| DeepSeek | 国内技术社区用户、开源爱好者 | 开源模型出圈，技术社区口碑传播，开发者浓度高 |
| Kimi (月之暗面) | 国内知识工作者、学生 | 主打长文档理解，用户多为需要处理文档/论文的群体 |
| Claude | 海外开发者、技术写作者 | 定位专业级 AI，用户偏高端技术群体，无内置搜索但有联网能力 |

## 二、对 MindSpore 社区的优先级判断

从开源 AI 计算框架社区的角度，按用户匹配度分层：

### 高优先级（技术用户集中）

- **Google AI Overview**：全球 AI 开发者的默认搜索入口
- **ChatGPT**：开发者问"怎么用 MindSpore 训练模型"最常用的地方
- **Perplexity**：技术选型对比场景（MindSpore vs PyTorch）
- **DeepSeek**：国内 AI 开发者首选

### 中优先级

- **Bing Copilot**：企业用户评估 AI 框架选型
- **Kimi**：用户上传 MindSpore 文档后提问
- **秘塔搜索**：学术/技术调研

### 低优先级

- **豆包**：用户群与 AI 框架开发者重叠度低
- **天工**：市场份额小

## 三、判断逻辑

1. **用户-产品匹配**：MindSpore 的目标用户是 AI 开发者和研究人员，优先关注开发者浓度高的平台
2. **搜索场景**：AI 框架相关问题多为"怎么做"（教程）和"选哪个"（对比），这类问题在 ChatGPT、Perplexity、DeepSeek 上最常见
3. **国内外分层**：MindSpore 同时面向国内和国际市场，需要覆盖中英文两个生态
4. **API 可用性**：半自动方案需要 API 支持，这也是一个筛选条件

## 四、API 可用性现状

| 平台 | API 搜索能力 | 备注 |
|------|-------------|------|
| Perplexity | 有 API，返回引用来源 | 最适合 GEO 分析 |
| ChatGPT | 有 API，支持 web search tool | 可获取搜索结果 |
| DeepSeek | 有 API，但无联网搜索 | 只能测知识覆盖，不能测搜索引用 |
| Google AI Overview | 无公开 API | 需要浏览器自动化 |
| Kimi | 有 API，支持联网 | 可测试 |

## 五、MVP 阶段结论

MVP 阶段聚焦 **Perplexity + ChatGPT + DeepSeek** 三个平台，覆盖三个维度：

- **Perplexity**：海外搜索，返回引用来源，最适合分析内容是否被引用
- **ChatGPT**：全球通用，用户基数最大，代表主流 AI 搜索行为
- **DeepSeek**：国内技术社区，代表国内开发者的 AI 使用习惯

三者均有 API 支持，可实现半自动化采样。

# 第二部分：关键词定义策略

问题来源分为**手动输入**和**自动生成**两条路径，最终合并为统一的问题集。

## 六、手动输入

社区运营根据经验手动编写问题，使用 Markdown 文档（`manual-questions.md`），skill 自动解析并转换为结构化 JSON。

### 输入格式（Markdown）

```markdown
# 手动问题集

## 了解阶段

- 主流深度学习框架有哪些？
- MindSpore 和 PyTorch 哪个更适合大模型训练？
- 国产 AI 框架有哪些？

## 使用阶段

- MindSpore 如何安装 GPU 版本？
- MindSpore 如何进行分布式训练？
- 如何将 PyTorch 模型迁移到 MindSpore？
```

Skill 读取后自动转换为 `questions.json` 中的结构化条目，自动推断分类和语言。

### 分类维度参考

| 类别 | 用户意图 | 示例 |
|------|---------|------|
| 认知 | 这是什么 | "MindSpore 是什么"、"国产 AI 框架有哪些" |
| 选型 | 选哪个 | "MindSpore vs PyTorch"、"MindSpore vs TensorFlow" |
| 教程 | 怎么做 | "MindSpore 安装"、"MindSpore GPU 训练配置" |
| 故障 | 出问题了 | "MindSpore 训练报错"、"MindSpore 算子兼容问题" |
| 特性 | 了解功能 | "MindSpore 自动并行"、"MindSpore Lite 端侧推理" |
| 迁移 | 怎么切换 | "PyTorch 迁移 MindSpore"、"MindSpore Converter 使用" |

## 七、自动生成

与步骤一的三条并行路径对应：

### 7.1 路径一：行业问题发现（了解阶段）

**目标**：生成用户在不知道 MindSpore 时可能搜索的行业大类问题。

**生成流程**：

1. LLM 确定社区领域层级：
   ```
   行业：人工智能
   细分：深度学习框架
   定位：全场景 AI 计算框架，支持端边云全场景部署
   竞品：TensorFlow、PyTorch、PaddlePaddle、JAX
   ```
2. 按用户意图分类生成问题：
   - **认知**："主流深度学习框架有哪些"、"国产 AI 框架有哪些"
   - **选型**："深度学习框架怎么选"、"大模型训练用什么框架好"
   - **趋势**："2026 年深度学习框架发展趋势"、"AI 框架国产化替代进展"
   - **场景**："端侧 AI 推理用什么框架"、"分布式训练框架对比"
3. 用竞品做反向扩展：
   - "PyTorch 有哪些替代品" → MindSpore 应被提及
   - "TensorFlow 和 PyTorch 之外还有什么框架" → MindSpore 应被提及

### 7.2 路径二：使用问题提取（使用阶段）

**目标**：从真实用户问题中提取高频使用类问题。

**数据源 A：Issue（Gitee/GitHub API）**

1. 拉取近 N 个月的 Issue 标题 + 标签
2. 过滤纯 Bug（如"段错误""编译失败"等不适合 GEO 的话题）
3. 聚类相似 Issue，提取高频主题
4. LLM 改写为用户可能搜索的自然语言问题

**转化示例**：

| 原始 Issue 标题 | 转化后的问题 |
|----------------|------------|
| `pip install mindspore fails on GPU` | "MindSpore GPU 版本安装失败怎么办" |
| `Support for dynamic shape in Graph mode` | "MindSpore 动态 shape 图模式支持" |
| 多个 Issue 提到 PyTorch 模型迁移问题 | "PyTorch 模型迁移到 MindSpore 教程" |

**数据源 B：论坛帖子（mindspore.cn 社区）**

1. 按浏览量/回复数排序，取 Top N 热门帖
2. 提取帖子标题和标签
3. LLM 改写为自然语言问题

### 7.3 路径三：AI 平台反向提取（使用阶段）

**目标**：从 AI 平台的视角发现用户常问的问题。

**生成流程**：

1. 向多个 AI 平台（Perplexity / ChatGPT / DeepSeek）提问：
   - "关于 MindSpore 最常被问到的问题有哪些？"
   - "用户在学习 MindSpore 时通常会遇到哪些问题？"
2. 收集各平台返回的问题列表
3. 取交集（多个平台都提到的问题优先级更高）

## 八、合并与去重

四条来源（手动 + 三条路径）的问题汇入统一问题集：

```
手动问题 ────────┐
路径一（行业）───┤
路径二（Issue/论坛）──┼──→ 合并 → 语义去重 → 分类标注 → 问题集
路径三（AI 平台）──┘
```

**去重策略**：
- 语义相似度 > 0.85 的合并为一个（用 LLM 判断）
- 手动问题优先级高于自动生成的
- 自动生成的按"出现频次 × 来源数量"排序

**输出**：`questions.json`（机器）+ `questions.md`（人工审阅）

## 九、技术实现要点

| 环节 | 方案 |
|------|------|
| 手动问题解析 | Skill 读取 `manual-questions.md`，LLM 自动推断分类和语言 |
| 行业问题生成 | LLM 根据社区领域层级 + 用户意图分类生成 |
| Issue 数据拉取 | Gitee API / GitHub API，按仓库批量获取 |
| 论坛数据拉取 | 论坛 API 或爬虫（需看论坛是否提供 API） |
| AI 平台反向提取 | 调用 Perplexity / ChatGPT / DeepSeek API |
| 语义去重 | LLM 判断语义相似度 |
| 问题存储 | JSON（机器）+ Markdown（人工审阅） |

# 第三部分：评分体系与输出规范（待讨论）

## 十、评分指标体系

分三个层级：

### 10.1 总览指标

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| GEO 综合评分 | 社区在 AI 搜索生态中的整体表现 | 各平台加权平均分（0-100） |
| 平台覆盖率 | 多少个平台能搜到社区内容 | 有提及的平台数 / 总平台数 |
| 周趋势 | 本周 vs 上周的评分变化 | 分数差值 + 箭头 |

### 10.2 平台维度（按平台展开）

每个平台（Perplexity / ChatGPT / DeepSeek）独立评分：

| 指标 | 说明 |
|------|------|
| 提及率 | 搜索关键词后，回答中提到社区的比例 |
| 引用率 | 回答中引用了社区官方链接的比例（仅 Perplexity 可精确统计） |
| 准确率 | AI 给出的信息是否正确、是否过时（由 LLM 自动判断） |
| 排名位置 | 社区内容在回答中出现的位置（首选推荐 / 对比提及 / 仅提一句） |
| 竞品对比 | 同一关键词下竞品（TensorFlow、PyTorch）的提及率 |

### 10.3 关键词维度（按关键词展开）

| 指标 | 说明 |
|------|------|
| 关键词覆盖状态 | 该关键词在各平台的搜索结果汇总 |
| 内容缺口标记 | AI 完全没提到社区 → 标记为"缺口" |
| 改进建议 | 针对该关键词的具体行动项 |

## 十一、GEO 评分体系

### 11.1 单关键词单平台评分（0-10 分）

```
评分 = 提及分 + 引用分 + 准确分 + 排名分

提及分（0-3）：
  0 = 完全未提及
  1 = 仅提及社区名称
  2 = 有实质性描述
  3 = 作为首选/重点推荐

引用分（0-3）：
  0 = 无引用来源
  1 = 引用第三方关于社区的内容
  2 = 引用社区官方内容但非最佳页面
  3 = 引用社区官方文档/博客的精准页面

准确分（0-2）：
  0 = 信息错误或严重过时
  1 = 基本正确但有小问题
  2 = 完全准确且为最新版本

排名分（0-2）：
  0 = 在竞品之后提及或仅做背景提及
  1 = 与竞品并列对比
  2 = 排在竞品之前或作为首选推荐
```

### 11.2 平台评分（0-100）

```
平台评分 = (所有关键词得分之和 / 关键词数 / 10) × 100
```

### 11.3 综合评分（0-100）

```
综合评分 = Perplexity 评分 × 0.35
         + ChatGPT 评分 × 0.40
         + DeepSeek 评分 × 0.25
```

权重依据：ChatGPT 用户基数最大权重最高，Perplexity 引用分析最精确次之，DeepSeek 覆盖国内市场。

### 11.4 评分等级

| 分数区间 | 等级 | 含义 |
|---------|------|------|
| 80-100 | A | GEO 表现优秀，AI 搜索生态覆盖充分 |
| 60-79 | B | 表现良好，部分关键词存在改进空间 |
| 40-59 | C | 表现一般，多个关键词存在内容缺口 |
| 20-39 | D | 表现较差，AI 平台对社区认知不足 |
| 0-19 | F | 基本缺失，需要系统性建设 |

## 十二、改进建议生成

根据评分结果，自动生成分优先级的改进建议。

### 建议示例

```
关键词："PyTorch 模型迁移到 MindSpore"
当前评分：4/10
问题诊断：
  - Perplexity：引用了第三方博客而非官方文档
  - ChatGPT：提到了 MindSpore Converter 工具但信息为旧版本
  - DeepSeek：未提及
改进建议：
  [P0] 更新官方迁移文档，确保包含最新 MindSpore Converter 版本信息
  [P1] 发布一篇结构化的迁移教程博客（含 FAQ 段落，便于 AI 引用）
  [P2] 在论坛置顶迁移专题帖，增加该话题的内容密度
```

### 优先级规则

- **P0**：评分 0-3 且为高优先级关键词 → 紧急补充内容
- **P1**：评分 4-6 → 优化现有内容
- **P2**：评分 7-8 → 微调优化
- 评分 9-10 → 无需行动，持续监控
