# get-question 与 scoring-engine 技能详解

> 面向新接触项目的人员，通俗介绍 GEO Search Assessment 项目中两个核心 skill 的设计思路、工作流程和输出物。

---

## 整体背景：这个项目在干什么？

想象一下：你是 MindSpore（一个 AI 框架）的运营人员。用户现在不去百度搜索了，而是直接问 ChatGPT、豆包、千问、DeepSeek 这些 AI 平台："MindSpore 怎么安装？"

问题来了 —— **这些 AI 平台的回答准确吗？有没有引用你的官方文档？还是在瞎编？**

这个项目就是一个**自动化诊断系统**，帮你回答这个问题，并且告诉你怎么改进。

整个流程是一条流水线，分 3 步：

```
get-question → platform-sampler → scoring-engine
  "问什么？"     "AI 怎么答的？"    "答得怎么样？怎么改？"
```

---

## get-question —— "问什么？"

### 一句话说明

这个 skill 负责**生成一份问题清单**——就是你要拿去问 AI 平台的那些问题。

### 为什么不直接手写问题？

手写当然可以，但你最多想到 10-20 个问题。真正的用户会问各种你想不到的问题。所以这个 skill 从 **4 个渠道**自动挖掘真实用户关心的问题，再加上你手写的，合在一起。

### 4 个问题来源（称为 path）

**Path 1：论坛（forum）—— 主力来源**

去 MindSpore 的 Discuss 论坛，自动爬取热门帖子。帖子标题往往是"MindSpore 多卡训练报错怎么办"这种，skill 会用 LLM 把它改写成更通用的搜索式问题："MindSpore 如何进行多卡分布式训练？"

**Path 2：Issue（issue）—— 主力来源**

去 GitCode 上的 MindSpore 仓库，爬取按评论数排序的 Issue。同样用 LLM 改写为搜索问题。比如 Issue 标题"TransData 算子输出格式不对"会变成"MindSpore TransData 算子支持哪些张量格式转换？"

**Path 3：邮件列表（maillist）**

从 MindSpore 的 SIG（特别兴趣小组）邮件列表中提取讨论话题。这些是核心开发者关心的问题，代表了深度用户的视角。

**Path 4：行业问题（industry）**

用 LLM 从行业角度生成问题。比如"MindSpore 和 PyTorch 相比有什么优势？""MindSpore 适合什么场景？"——这些是**了解阶段**的用户会问的。

### 手写问题

你也可以在 `manual-questions.md` 文件里直接写问题。skill 会解析它，和自动生成的合并。

### 合并去重

4 个渠道 + 手写的问题汇总后，LLM 会做语义去重（"怎么安装 MindSpore"和"MindSpore 安装方法"是同一个问题），最终输出 30-40 个不重复的问题。

### 输出

- `questions.json` —— 结构化数据，给下一步 platform-sampler 用
- `questions.md` —— 人类可读版本，给你审核用

### 人工审核

最后会**暂停**，让你看看生成的问题列表，删掉不合适的，补上遗漏的。通过后才进入下一步。

### 关键配置

| 参数 | 说明 | 例子 |
|------|------|------|
| `community` | 目标社区名 | `MindSpore` |
| `paths` | 用哪些渠道 | `forum,issue` 或 `all` |
| `limit` | 每个渠道爬多少条 | `50` |
| `forum_url` | 论坛地址 | `https://discuss.mindspore.cn` |

### 流程图

```
manual-questions.md ─────┐
                         │
论坛热门帖子 ──LLM改写──┐ │
                       │ │
GitCode Issue ─LLM改写─┤ ├──▶ 合并 ──▶ 语义去重 ──▶ questions.json
                       │ │                          questions.md
SIG 邮件列表 ──LLM改写─┤ │                              │
                       │ │                              ▼
行业问题 ──LLM生成─────┘ │                        ⏸ 人工审核
                         │
                         ┘
```

## scoring-engine —— "答得怎么样？怎么改？"

### 一句话说明

这个 skill 拿到 AI 平台的回答后，**逐条分析每个回答的质量**，判断属于哪种"症状"，然后从一个 72 条建议的标准库中匹配最适合的优化建议。

### 输入是什么？

在 scoring-engine 之前，还有一个 `platform-sampler` skill（这里不展开），它会拿着 get-question 生成的问题去问 4 个 AI 平台，把回答收集到 `responses.json`。

scoring-engine 需要的输入：

| 文件 | 谁提供 | 内容 |
|------|--------|------|
| `responses.json` | platform-sampler 自动生成 | 每个问题在每个平台的回答原文 |
| `content-labels.json` | **人工标注** | 每个问题在官网上有没有对应内容 |
| `standard-answers.json` | 可选，人工准备 | 官方标准答案（有了可以做更深的分析） |

### 分析过程：逐层深入

#### Layer 1：官网有没有这个内容？（人工判断）

最基本的问题：用户问的这个事情，官网上有没有写过？

- 如果**没写过**（比如用户问"MindSpore 的版本发布策略是什么"，但官网上根本没有这个文档）→ 标记为**现象 A（内容缺口）**，优先级 P0（最紧急），因为 AI 巧妇难为无米之炊。
- 如果**写过** → 进入 Layer 2 继续分析。

#### Layer 2：AI 回答引用官方了吗？引对了吗？（LLM 判断）

对于官网有内容的问题，让 LLM 分析 AI 平台的回答，回答 4 个核心问题：

1. AI 的回答引用了哪些来源？
2. 这些来源里有多少是官方的？（计算官方引用比例）
3. 引用的信息是否准确？
4. 回答存在哪些具体的内容问题？

根据分析结果，归入 4 种"症状"之一：

| 现象 | 通俗解释 | 严重程度 | 举例 |
|------|---------|---------|------|
| **B — 有内容没引用** | 官网明明写了，但 AI 完全没看到 | P1 | 官网有安装文档，但 ChatGPT 回答时没提到官网 |
| **C — 引用了但说错了** | AI 引用了官方，但信息不对 | P0（最严重） | 豆包说"MindSpore 支持直接读取 PyTorch 模型"，但实际不支持 |
| **D — 引用充分且准确** | 很好，不用管 | 无需行动 | 4 个平台回答安装方式都正确且引用了官网 |
| **E — 引用了但比例低** | 官方被提了一嘴，但大部分内容来自第三方博客 | P1 | 官方引用占 20%，其他 80% 来自 CSDN 博客 |

加上 Layer 1 的现象 A，一共就是 **A/B/C/D/E 五种现象**，覆盖了所有可能的情况。

同时，LLM 还会给每个回答打上**问题标签**（26 种），比如：

- `negation_reversed` —— "不支持"被说成了"支持"
- `fabricated_claims` —— AI 编造了不存在的功能
- `ambiguous_terminology` —— 术语和其他产品搞混了
- `no_direct_answer` —— 回答绕了半天没给结论
- `outdated_info` —— 引用了过时的版本信息
- `shallow_content` —— 内容太浅，缺乏深度

这些标签的作用是**连接"诊断"和"建议"**——后面会根据标签从建议库中精准匹配改进措施。

#### Layer 2+：事实覆盖检查（可选加强版）

如果你额外准备了标准答案（`standard-answers.json`），这一层会做更精细的分析：

1. 先从标准答案中提取**关键事实**清单，比如：
   - POSITIVE："支持 Ascend/GPU/CPU 三种硬件"
   - NEGATIVE："不支持直接读取 PyTorch 的 .pth 文件"

2. 然后逐条检查 AI 回答是否覆盖了这些事实：
   - **covered** —— 正确传达了
   - **missed** —— 遗漏了
   - **contradicted** —— 说反了（最危险！）
   - **fabricated** —— 编造了标准答案里没有的东西

这一层特别擅长发现**否定信息被反转**的问题。比如官方说"不支持 X"，AI 却说"支持 X"。

### 跨平台模式分析

单独看每个回答容易只见树木不见森林。这一步把所有平台的结果放在一起看，回答一个关键问题：**这个问题是某个 AI 平台的毛病，还是我们自己官网内容的问题？**

判断规则很简单：

- 如果同一个错误只在 1 个平台出现 → **平台侧问题**（那个平台的模型有偏差）
- 如果同一个错误在 3 个以上平台出现 → **内容源问题**（说明是官网内容本身不够清晰，导致多个 AI 都理解错了）

**内容源问题**会被自动提升优先级，因为改官网能一次性影响所有平台。

此外，这一步还会识别出几种规律性问题：

| 模式 | 说明 | 举例 |
|------|------|------|
| 幻觉模式 | 哪些话题容易让 AI 编造内容 | "TransData"这个词让 3 个平台都误以为是大数据 ETL 工具 |
| 消歧义问题 | 哪些术语被和其他产品搞混 | "数据转换"被混淆为大数据领域的数据转换 |
| 否定传递失败 | 哪些"不支持"被一致性地说成了"支持" | "不支持直接读取 PyTorch 模型"被 3 个平台说反 |
| 引用盲区 | 哪些话题在所有平台都没有引用官方 | "社区贡献流程"在 4 个平台都没引用官网 |

### 建议匹配：从 72 条建议库中精准挑选

这是和之前最大的改进。以前建议是 LLM 随机生成的，每次说法不一样，质量不稳定。

现在有一个预定义的**72 条 GEO 优化建议目录**（`geo-suggestions-catalog.md`），覆盖 12 个维度：

| 维度 | 数量 | 解决什么问题 |
|------|------|-------------|
| CTX 上下文清晰度 | 9 条 | 回答开头不直接、术语没定义、没覆盖追问... |
| ORG 组织结构 | 9 条 | 没有表格、没有 Schema 标记、段落太长... |
| REF 可引用性 | 8 条 | 数字模糊、没有时间戳、引用链断裂... |
| EXC 独占性 | 10 条 | 内容太浅、没有原创数据、竞品说得更好... |
| DIS 消歧义 | 4 条 | 术语和别的产品搞混了 |
| NEG 否定强化 | 5 条 | "不支持"写得不够醒目 |
| VER 版本 SEO | 3 条 | AI 引用了过期文档 |
| KWD 关键词覆盖 | 3 条 | 用户换个说法就找不到了 |
| EXP/EPT/AUT/TRU | 15 条 | 经验/专业性/权威/信任信号 |
| SITE 站点级 | 6 条 | 爬虫被屏蔽、SPA 空白页、sitemap 过期... |

匹配过程分 3 步：

1. 根据**现象类型**（A/B/C/D/E）缩小范围 → 比如现象 C 只看 22 条相关建议
2. 根据**问题标签**（`issues_found`）精准匹配 → 比如标签是 `negation_reversed` → 匹配 NEG-01、NEG-02、NEG-04
3. 每个问题最多选 5 条最相关的建议，多个平台出现相同问题的建议会合并

### 最终输出

**`scoring-results.json`** —— 完整的结构化数据，包含：

- 每个（问题, 平台）的评分、现象分类、问题标签
- 跨平台模式分析结果
- 匹配到的建议列表

**`suggestions.md`** —— 人类可读的报告，包含 10 个章节：

| 章节 | 内容 |
|------|------|
| 一、概览 | 多少问题、多少平台、平均引用比例 |
| 二、核心发现 | 最重要的 3 个发现 |
| 三、现象分布 | A/B/C/D/E 各有多少 |
| 四、严重级别 | P0/P1/P2 各有多少 |
| 五、平台对比 | 哪个平台表现好、哪个差 |
| 六、跨平台模式 | 幻觉、消歧义、否定传递等规律 |
| 七、改进建议 | 按 P0/P1/P2 分组的具体建议 |
| 八、逐题详情 | 每个问题的详细分析 |
| 九、执行路线图 | 1-2 周 / 2-4 周 / 长期的行动计划 |
| 十、KPI 跟踪 | 官方引用率、事实准确率等目标值 |

### 人工校准（学习循环）

最后会暂停，建议你抽查 20% 的评分结果。如果你发现 LLM 的判断有偏差，记录到 `scoring-calibration.md`，下次运行时这些校准信息会被自动纳入 prompt，让 LLM 避免同样的错误。这是一个**学习循环**。

## 两个 skill 的关系

```
get-question                    scoring-engine
┌─────────────────┐            ┌──────────────────────────────┐
│ 输入：社区名称    │            │ 输入：AI 回答 + 人工标注       │
│                 │            │                              │
│ 做什么：          │  ──JSON──▶ │ 做什么：                      │
│ 从4个渠道挖掘问题  │            │ 1. 官网有没有这个内容？         │
│ + 合并去重        │            │ 2. AI 引用对了吗？（5种现象）   │
│                 │            │ 3. 事实覆盖检查（可选）         │
│ 输出：问题清单    │            │ 4. 跨平台找规律               │
│                 │            │ 5. 从72条建议库中匹配建议       │
│ 暂停：人工审核    │            │                              │
└─────────────────┘            │ 输出：评分报告 + 改进建议       │
                               │ 暂停：人工抽查校准              │
                               └──────────────────────────────┘
```

中间还有一个 `platform-sampler`（这里没展开），它负责拿着问题去问 AI 平台、收集回答。三个 skill 串起来就是完整的诊断流程。

